# 일관성 있는 협력

재사용은 공짜로 얻어지지 않음 -> 객체들의 협력 방식을 일관화시켜야 한다



코드 재사용을 위한 상속은 지양하여야 함

### 설계에 일관성 부여하기

* '변하는 개념'을 '변하지 않는 개념' 으로부터 분리하기
* 변하는 개념을 캡술화하기
  * 캡슐화는 단순한 데이터 은닉을 넘어 변하는 모든 것을 은닉하는 것임
  * 즉 자주 변경되는 내부 구현 또한 퍼블릭 인터페이스 뒤로 은닉해야 함
  * 협력 패턴을 만들어서 맞추어라



## 질문

* Q1. 코드 재사용을 위하여 상속을 이용하면 무슨 문제가 발생하는가?
  * 실제 포함 관계가 아닌데도 단순히 메서드 재사용을 위해 상속을 이용하게 되어 결합을 만들면 추후 재설계가어려움. 조합으로 바꾸어야 함
* Q2. 캡슐화에 대하여 이번 장에 나온 내용을 기반으로 설명
  * 캡슐화란 데이터만 숨기는 것이 아니라 자주 변하는 구현을 인터페이스 뒤로 숨기는 것
